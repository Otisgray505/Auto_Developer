# Autonomous CLI Orchestration: Deep Research & Architecture Solution

## 1. Executive Summary
This research report outlines the technical architecture for an **Autonomous Workflow Engine** driven by the **Antigravity Framework**. The primary objective is to enable Antigravity to act as the central "Brain" (Project Manager), securely and efficiently orchestrating CLI-based AI tools like **Claude Code**, **Codex**, and **Gemini CLI** through advanced networking techniques including **Proxy Wrapping** and **API Gateways**.

## 2. Demystifying Core Concepts: Proxies & Proxy Wrapping

The user requested an explanation of "Proxy" and "Proxy Wrapping" (specifically Antigravity Proxy), their dependency on projects, and obsolescence.

*   **Proxy (Forward/Reverse):** A server application that acts as an intermediary for requests from clients seeking resources from other servers. It operates at the network layer. It is a fundamental, universal internet technology—**not obsolete**, and entirely independent of any specific project.
*   **Proxy Wrapping (DLL Injection / Hooking):** As seen in the `yuaotian-antigravity-proxy` repository, this is a highly specialized technique. It forces a specific application to route its traffic through a proxy without changing the system-wide OS proxy settings. It does this by injecting a custom DLL (like hijacking `version.dll`) into the target process's memory space to intercept network API calls (like `winsock`).
    *   **Is it obsolete?** No, it is actively used in advanced routing, game anti-cheat, and reverse engineering.
    *   **Is it dependent on the project/OS?** **YES. Extremely.** DLL injection is strictly tied to Windows architecture (PE format). This is exactly why it **fails in WSL (Windows Subsystem for Linux)**, as Linux uses ELF binaries and different networking stacks.

## 3. High-Level Architecture Design

To achieve the vision where Antigravity governs the entire project, avoids hallucinations, and delegates tasks bidirectionally, we must implement a **Hub-and-Spoke Architecture** with strict network and permission boundaries.

### The Components

1.  **The Brain (Antigravity Orchestrator + BMAD Master)**
    *   Acts as the unified Project Manager. Runs locally or on a secure remote server. Maintains persistent context, project state, and memory.
2.  **The API Gateway (router-for-me-CLIProxyAPI)**
    *   The central nervous system. Instead of CLI tools directly calling the internet, all traffic routes through this Go-based gateway.
    *   Controls quotas, enforces TLS, caches responses, and routes Claude Code/Codex requests to the appropriate backend LLM (Anthropic, OpenAI, or local models).
3.  **The Execution Layer (Edge Executors)**
    *   Claude Code, Gemini CLI, and custom agents operate inside isolated **Docker Containers** or **WSL instances**.
    *   Since DLL proxy wrapping fails in WSL, these containers are configured via standard Environment Variables (`HTTP_PROXY`, `HTTPS_PROXY`) pointing to the CLIProxyAPI.
4.  **The Interaction Bridges (Remote Access)**
    *   **Telegram Bot Bridge:** Enables the user to query status, approve PRs, or inject new context remotely from a mobile device without keeping the browser open (requires migration from browser-script to a Node.js background service).
    *   **ClawX / OpenClaw:** Serves as the rich desktop GUI dashboard, allowing real-time monitoring of agent logs, task graphs, and API usage stats via OpenClaw Gateway.

## 4. Autonomous Workflow Orchestration: Step-by-Step

How the system functions actively to keep the project on track:

### Phase 1: Ingestion & Planning
1. User sends a command via the **Telegram Bot Bridge** or **Antigravity CLI**: *"Implement a new Redis caching layer."*
2. The **Antigravity Orchestrator** receives this. It utilizes `/brainstorming-coach` and `/analyst` skills to decompose the request into a robust `task.md`.

### Phase 2: Secure Delegation (The "Spawning")
1. The Orchestrator determines that **Claude Code** is best suited for the heavy refactoring involved in the caching layer.
2. It spins up an ephemeral Docker container.
3. **Privilege Control:** The container is assigned a temporary, scoped API Key generated by **CLIProxyAPI**. This key has a hard spending limit ($1.00) and is restricted to only use the `Claude-3.7-Sonnet-thinking` route.

### Phase 3: Execution & Bidirectional Feedback
1. Inside the container, Claude Code runs its autonomous loop. Every LLM request it makes hits the **CLIProxyAPI**.
2. If Claude Code encounters an architectural ambiguity, it outputs a specific strict-formatted log.
3. The **Antigravity Orchestrator** parses this standard output. If human intervention is needed, it forwards the query to the user via the **Telegram Bot Bridge**.
4. The user replies via Telegram; the Orchestrator feeds the response back to Claude Code's `stdin`.

### Phase 4: Verification & Integration
1. Claude Code finishes. The Orchestrator spins up a secondary agent (`/testarch-automate`) to run tests.
2. If tests pass, the code is merged. The scoped API key is revoked by the CLIProxyAPI.

## 5. Why This Approach Prevents AI Degradation (Hallucinations)

1.  **Strict State Governance:** CLI tools like Claude Code are excellent at coding but terrible at long-term project management. Antigravity handles the "Meta-State" (PRDs, Epics, Sprint Plans), while Claude Code handles the "Micro-State" (file modifications).
2.  **Network-Level Restraints:** By wrapping everything behind the **CLIProxyAPI**, we prevent rogue agents from burning API credits in infinite loops.
3.  **Ephemeral Sandboxes:** Agents execute in isolated Docker/WSL environments. If an agent goes off-track, the Orchestrator kills the container and restarts from the last known good state.

## 6. Implementation Roadmap Options

### Option A: Fully Local (Maximum Privacy, Moderate Complexity)
*   **Networking:** Use local TUN mode (e.g., sing-box) to enforce global routing for WSL and Windows.
*   **Execution:** Run Antigravity locally connected to a locally hosted CLIProxyAPI routing to local Llama3 instances.
*   **Remote:** Requires setting up a sophisticated reverse tunnel (Cloudflare Tunnels) or ngrok for Telegram/Web access.

### Option B: Cloud-Hybrid (High Efficiency, Recommended)
*   **Networking:** Host the `CLIProxyAPI` on a secure VPS (e.g., Odoo Backend Server space).
*   **Execution:** Antigravity runs locally/securely in cloud. It spawns CLI tools via ssh or docker API over mTLS.
*   **Remote:** Node.js Telegram Bot running on the VPS holding persistent WebSocket connection to the Orchestrator.
*   **ClawX UI:** Connects to the local OpenClaw gateway to visualize the entire node execution graph.
